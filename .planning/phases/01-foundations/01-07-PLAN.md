### src/instrumentation.ts (RESEARCH.md Pattern 9 lines 680–691 — Next.js built-in hook)
```ts
import * as Sentry from '@sentry/nextjs';

export async function register() {
  if (process.env.NEXT_RUNTIME === 'nodejs') {
    await import('../sentry.server.config');
    // D-12 bootstrap admin on Node cold start
    const { bootstrapAdmin } = await import('./lib/bootstrap');
    try {
      await bootstrapAdmin();
    } catch (err) {
      Sentry.captureException(err, { tags: { phase: 'bootstrap-admin' } });
    }
  }
  if (process.env.NEXT_RUNTIME === 'edge') {
    await import('../sentry.edge.config');
  }
}

export const onRequestError = Sentry.captureRequestError;
```
**Why this file lives in `src/`:** Next.js 16 accepts `instrumentation.ts` at the repo root OR at `src/instrumentation.ts`. Since the project uses `--src-dir`, we keep it under `src/` alongside other app code.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Sentry SDK → third-party | PII is scrubbed server-side before leaving the Vercel boundary |
| bootstrap boot hook → DB | Runs with the same privileges as any Node request (pooled DATABASE_URL) |
| /api/_smoke/sentry → external attackers | Smoke endpoint must be admin-gated so attackers can't spam Sentry quota |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-BOOTSTRAP-01 | Spoofing (bootstrap admin takeover) | `src/instrumentation.ts` + `src/lib/bootstrap.ts` | mitigate | Instrumentation hook calls `bootstrapAdmin()` which uses `.onConflictDoNothing()` on email PK. Even if `BOOTSTRAP_ADMIN_EMAIL` is left set after real admins exist, the insert is a no-op (email PK conflict). Error captured to Sentry via try/catch so a DB-down scenario doesn't hang Node cold start. Verified by `tests/db/auth-signin-callback.test.ts` (plan 05) + the smoke test in Task 07.3 (first cold start after plan 03 migration creates the bootstrap row). |
| T-SEC-ENV | Information Disclosure (client-exposed secret via Sentry) | `sentry.client.config.ts` | mitigate | Client config uses `NEXT_PUBLIC_SENTRY_DSN` (intended-public; see Sentry docs — DSNs are not secrets). `SENTRY_AUTH_TOKEN` (which IS a secret) is referenced only in `next.config.mjs` and consumed by the build-time `withSentryConfig` to upload source maps; never shipped to client. `hideSourceMaps: true` further ensures uploaded source maps aren't publicly accessible. |
| T-SMOKE-ABUSE | Denial of Service (Sentry quota exhaustion) | `src/app/api/_smoke/sentry/route.ts` | mitigate | Endpoint calls `requireAdmin()` first — unauthenticated callers get 401 and never reach the throw, so no Sentry event is emitted. Task 07.3 only intended for one-shot manual verification. |
</threat_model>
