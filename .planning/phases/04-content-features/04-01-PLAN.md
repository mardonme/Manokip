<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Migration runner → Neon dev branch | DDL applies to a managed Postgres instance; failure mode is connection error or DDL syntax rejection (caught by drizzle-kit migrate exit code) |
| Verifier script → information_schema | Read-only system-catalog probe; no privilege escalation surface |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01-01 | Tampering | Hand-authored migration SQL drift from drizzle-kit-generated SQL | mitigate | All schema-driven DDL goes through `drizzle-kit generate` (auto). Only the backfill UPDATE + CHECK CONSTRAINT + CREATE VIEW are hand-authored — these are not drizzle-kit's responsibility. Verifier script asserts the final state regardless of how it was authored (information_schema is the SSOT). |
| T-04-01-02 | Information Disclosure | pgView leaking draft content via UNION ALL without WHERE filter | mitigate | View body explicitly filters WHERE r.status='published' AND WHERE i.status='published' — same posture as Phase-2 product_translation_completeness pgView (defense-in-depth). 7-th spec asserts draft rows are absent from view output. |
| T-04-01-03 | Tampering | Migration applied with no backfill leaving NULL status on existing rows | mitigate | UPDATE backfill statement runs in the same migration file BEFORE the CHECK constraint is added. If backfill is somehow skipped, the CHECK constraint addition fails on NULL rows (Postgres rejects). At Phase 4 open, recipe + industry tables have 0 rows so backfill is a no-op — verifier still asserts the column shape end-to-end. |

</threat_model>
