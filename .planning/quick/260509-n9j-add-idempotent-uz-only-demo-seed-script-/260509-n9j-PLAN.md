### product_search uz rebuild SQL — VERBATIM transcription contract

Source: `src/actions/products.ts:113-133`. The seed must reproduce the uz-only branch
(spec_text agg returns NULL because we insert zero spec values — that's fine, `coalesce(..., '')` handles it):

```sql
INSERT INTO product_search (product_id, locale, search_tsv)
SELECT $1::uuid, 'uz'::text,
  setweight(to_tsvector('simple'::regconfig, coalesce(t.name, '')), 'A') ||
  setweight(to_tsvector('simple'::regconfig, coalesce(t.short_desc, '')), 'B') ||
  setweight(to_tsvector('simple'::regconfig, coalesce(t.long_desc, '')), 'C') ||
  setweight(to_tsvector('simple'::regconfig, coalesce('', '')), 'D')
FROM product_translations t
WHERE t.product_id = $1::uuid AND t.locale = 'uz'::text
ON CONFLICT (product_id, locale) DO UPDATE SET search_tsv = EXCLUDED.search_tsv
```

Note: simplified the spec aggregation to literal `''` (no spec values exist for demo
products — confirmed by Phase 1/2 schema; the lateral join in production is for
products that DO have spec values).

### audit_log decision

The seed does NOT write audit_log rows. Justification:
- `audit_log` columns (`actor_email`, `action`, `entity_type`, `entity_id`,
  `before_json`, `after_json`) are all nullable; the table only enforces
  `id` (bigserial) and `at` (default now). There is NO database-level
  trigger or constraint requiring audit rows on insert. Audit is a
  Server-Action-layer convention, not a schema invariant.
- The seed is one-off scaffolding; auditing it would muddy the audit trail
  with synthetic actor entries.
- This deviation is documented in the seed file's header comment and in
  scripts/README.md.

### revalidateTag decision

The seed does NOT call `revalidateTag` — it has no Next.js runtime. This
matches the CLAUDE.md guardrail interpretation: that guardrail is for
SERVER ACTIONS. CLI scripts cannot honor it because there's no Next runtime
to invalidate. Documented in seed header. The user must restart `pnpm dev`
or hit a page once after seeding to see results (RSC ISR re-renders on next
request in dev).

6. **Categories — insert-if-missing (idempotent upsert)**:
   For each category, the strategy is:
   a. Try to find an existing category by uz slug:
      ```ts
      const existing = await tx
        .select({ id: categoryTranslations.categoryId })
        .from(categoryTranslations)
        .where(sql`locale = 'uz' AND slug = ${cat.slug}`)
        .limit(1);
      if (existing.length > 0) {
        catIdBySlug[cat.slug] = existing[0].id;
        continue; // skip — admin or prior seed already created it
      }
      ```
   b. Otherwise insert the category and its uz translation:
      ```ts
      const [row] = await tx.insert(categories).values({
        parentId: null, sortOrder: cat.sortOrder,
      }).returning({ id: categories.id });
      await tx.insert(categoryTranslations).values({
        categoryId: row.id, locale: 'uz', name: cat.name, slug: cat.slug,
      });
      catIdBySlug[cat.slug] = row.id;
      ```
   Rationale for select-then-insert (instead of one-shot
   `.onConflictDoNothing()`): the parent `categories` row uses
   `defaultRandom` UUIDs — there's no natural conflict key on the parent.
   The conflict key is on the translation. Doing select-first lets us
   surface the existing parent ID for the products-insert pass.

7. **Manufacturers — same select-then-insert pattern as categories**:
   key on `manufacturerTranslations` uz slug. Insert manufacturer base row
   only if no existing translation matches. Use `websiteUrl: mfr.websiteUrl,
   isOfficialRep: false, logoPublicId: null` on insert.

8. **Products insert** — in a `for (const p of DEMO_PRODUCTS)` loop:

     longDesc: null,
